---
name: evolve-agents
description: 从Agent会话中提炼经验并更新Agent配置，使其自进化。发现Agent反复犯错、踩坑、遗漏时调用。
---

## 一、何时进化

满足任一条件即可调用 `/evolve-agents`：

- Agent **反复犯同样的错误**（如代码审计师报错后开发 agent 修复后又犯）
- 发现了**Agent 配置中未提及的隐藏前置条件或陷阱**
- Agent **忽略了项目特有的规范**（如命名、格式、响应格式）
- 用户明确纠正了 Agent 的行为（"不要这样"、"以后应该这样"）
- 一个 Agent 的输出与下一个 Agent 的预期**不匹配**

**不满足则跳过**：无实际经验积累不强行进化，不要创建空壳。

---

## 二、进化什么

用 ROI 框架判断是否值得更新 Agent 配置：

```
ROI = (Agent忽略此信息后的犯错概率 × 犯错代价) / 更新配置的维护成本
```

ROI > 0 时值得更新。具体来说，必须同时满足：

1. **实际会话中发生过的**（不是推理、猜测、未执行的计划）
2. **未来同类任务还会遇到的**（不是一次性场景）
3. **当前 Agent 配置未覆盖的**（已写了的不要重复）

### 值得进化的内容

| 类型 | 说明 | 写入位置 |
|------|------|---------|
| 项目特有的隐藏坑点 | 不写就会再犯的隐性规则 | Agent 的「关键坑点」章节 |
| Agent 间信息传递缺陷 | 上游输出与下游预期不匹配 | 相关 Agent 的「输入文档」或「工作流程」 |
| 项目规范遗漏 | Agent 产出的代码不符合项目惯例 | Agent 的「项目上下文」章节 |
| 新增的有效模式 | 多轮试错后发现更优做法 | Agent 的「已验证方案」章节 |
| Skill 引用调整 | 发现某个 Skill 实际没用或缺少某个 Skill | Agent 的「可用 Skills」章节 |

### 禁止进化的内容

| 禁止 | 原因 |
|------|------|
| 未验证信息 | 推理、猜测、"应该是"、"可能可以" |
| 易变状态 | 临时路径、PID、session ID |
| 通用常识 | AI 零样本就能推断的知识 |
| 冗余重复 | Agent 配置中已有覆盖的内容 |
| 修改不可变属性 | name、model、color、工具权限、返回状态定义、过程日志格式 |

## 三、怎么写进化内容

### 新增章节格式

**关键坑点**（新增到 Agent 配置中）：

```markdown
## 关键坑点

- ⚠致命 {坑点描述}（不写就会导致崩溃/数据丢失）
- ⚠隐蔽 {坑点描述}（不报错但结果错误，很难发现）
- ⚠易忘 {坑点描述}（不提醒就漏掉）
```

**已验证方案**（新增到 Agent 配置中）：

```markdown
## 已验证方案

- {方案描述}（经过实际执行验证）
```

### 更新项目上下文

如果项目新增了模块、接口、规范，追加到 Agent 的「项目上下文」章节中：

```markdown
### 新增模块
| 模块 | 路由前缀 | 说明 |
|------|---------|------|
| {新模块} | {路由} | {描述} |

### 新增规范
- {规范描述}
```

### 更新 Agent 间信息传递

如果上游 Agent 输出格式与下游不匹配，更新上下游 Agent 的输入/输出描述。

## 四、怎么落地

### 步骤

1. **回顾会话**：回顾当前会话中所有 Agent 的执行记录（日志、输出文件）
2. **识别进化点**：按 ROI 框架筛选值得进化的内容
3. **定位目标 Agent**：确定应该更新哪个 Agent 配置文件
4. **读取当前配置**：用 Read 工具读取 `.claude/agents/{agent-name}.md`
5. **更新配置**：用 Edit 工具局部修改 `.claude/agents/{agent-name}.md`（新增章节或追加内容）
6. **验证更新**：确认修改内容正确，不影响 Agent 的不可变属性

### 规则

1. **先查已有**：更新前先 Read 当前配置，检查是否已有覆盖
2. **无新跳过**：已有覆盖的、无新增价值的 → 不操作
3. **用 Edit 局部修改**：禁止用 Write 覆盖整个 Agent 配置文件
4. **追加而非替换**：在已有章节中追加内容，不删除已有内容
5. **只更新部署文件**：只修改 `.claude/agents/*.md`，不修改源模板 `_agents/*.md`

> 错误的修改每轮都会被加载并放大。宁可不动，不要改错。

## 五、示例

### 示例 1：后端 Agent 忽略了项目统一响应格式

**场景**：后端开发 产出的 API 返回自定义格式，但项目规范要求统一的响应格式。

**进化操作**：

读取 `.claude/agents/backend-developer.md`，在「项目特有规范」章节追加：

```markdown
- ⚠致命 所有 API 必须使用项目统一的响应格式，禁止自定义响应结构
```

### 示例 2：前端 Agent 没检查权限

**场景**：前端开发 新建的页面缺少权限检查，整体测试 发现后开发者修了两次。

**进化操作**：

在 `.claude/agents/frontend-developer.md` 的「关键坑点」章节（如没有则新增）追加：

```markdown
## 关键坑点

- ⚠易忘 所有业务页面必须检查权限，新建页面时不要遗漏
```

### 示例 3：开发代表 输出格式与开发 agent 预期不匹配

**场景**：开发代表 输出的任务列表用 markdown 表格，但开发 agent 期望 JSON 格式的任务 ID 列表。

**进化操作**：

更新 `.claude/agents/planner.md` 的返回格式章节，在 tasks 字段中增加 `id` 和 `description`；同时更新 `.claude/agents/backend-developer.md` 和 `.claude/agents/frontend-developer.md` 的「工作流程」章节，明确说明如何从开发代表结果中提取任务列表。

### 示例 4：代码审计师 发现 UI 组件库的隐蔽问题

**场景**：代码审计师 的构建通过了，但前端 Agent 使用了某个 UI 组件的特定属性导致显示问题，整体测试 发现后才修复。

**进化操作**：

在 `.claude/agents/frontend-developer.md` 和 `.claude/agents/verifier.md` 的项目上下文中追加：

```markdown
## 关键坑点

- ⚠隐蔽 {UI 组件库} 的 {特定属性} 会导致 {问题描述}，禁止使用
```

## 六、与 my-agents-info 的关系

`/evolve-agents` 和 `/my-agents-info` 的区别：

| | `/my-agents-info` | `/evolve-agents` |
|---|---|---|
| 触发时机 | 项目初始化或手动全量同步 | 发现 Agent 犯错后按需触发 |
| 更新范围 | 扫描整个项目文档，全量更新所有 Agent | 只更新出问题的 Agent |
| 更新内容 | 项目上下文（架构、技术栈、业务规则） | 关键坑点、已验证方案、信息传递修复 |
| 更新方式 | 扫描 → 对比 → 更新 | 回顾会话 → 识别 → 追加 |

两者互补：`/my-agents-info` 做全量初始化，`/evolve-agents` 做增量自进化。
