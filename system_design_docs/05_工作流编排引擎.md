### 5.1 状态定义

```python
from typing import TypedDict, List, Any, Literal
from langgraph.graph import MessagesState

class AgentState(TypedDict):
    """LangGraph共享状态"""
    user_input: str                          # 用户原始输入
    messages: List[Any]                      # 对话历史
    current_agent: str                       # 当前执行的agent
    next_agent: str                          # 下一个agent（用于路由）
    intermediate_results: dict               # 中间结果存储 {agent_name: result}
    final_answer: str                        # 最终答案
    need_human: bool                         # 是否需要人工介入
    human_input: str                         # 人工输入
```

### 5.2 监督者模式实现

```python
from langgraph.graph import StateGraph, END
from langgraph.checkpoint import MemorySaver

def create_supervisor_workflow(supervisor_agent, worker_agents):
    """创建监督者模式工作流"""
    graph = StateGraph(AgentState)
    
    # 添加主管节点
    graph.add_node("supervisor", supervisor_agent)
    
    # 添加工作节点
    for name, agent in worker_agents.items():
        graph.add_node(name, agent)
        # 工作节点执行完后返回主管
        graph.add_edge(name, "supervisor")
    
    # 条件路由：主管决定下一个节点
    def route_supervisor(state: AgentState) -> Literal[*worker_agents.keys(), "end"]:
        next_agent = state.get("next_agent", "end")
        if next_agent == "end" or next_agent not in worker_agents:
            return "end"
        return next_agent
    
    graph.add_conditional_edges("supervisor", route_supervisor)
    graph.set_entry_point("supervisor")
    
    return graph.compile(checkpointer=MemorySaver())
```

### 5.7 工作流错误处理策略

#### 5.7.1 策略定义

每个工作流支持三种错误处理策略，在 `workflows` 表中通过 `error_strategy` 字段配置：

| 策略 | 值 | 行为 |
|------|------|------|
| **fail-fast** | `"fail_fast"` | 任一 Agent 失败，工作流立即终止，返回已收集的中间结果 |
| **skip-and-continue** | `"skip_continue"` | 失败 Agent 被跳过，标记为失败，后续 Agent 继续执行 |
| **retry** | `"retry"` | 失败 Agent 自动重试（最多 N 次），全部失败后再按 fail-fast 处理 |

#### 5.7.2 超时控制

```python
from dataclasses import dataclass
from typing import Optional
import asyncio

@dataclass
class WorkflowTimeoutConfig:
    """工作流超时配置"""
    agent_timeout_seconds: int = 60      # 单个 Agent 超时（秒）
    workflow_timeout_seconds: int = 300  # 整个工作流超时（秒）
    retry_count: int = 2                 # 重试次数
    retry_delay_seconds: int = 3         # 重试间隔（秒）

class WorkflowExecutor:
    """带错误处理的工作流执行器"""
    
    def __init__(self, config: WorkflowTimeoutConfig):
        self.config = config
        self.error_log: list[dict] = []  # 错误日志
    
    async def execute_with_timeout(self, agent_fn, state: dict, 
                                    agent_name: str) -> dict:
        """带超时的 Agent 执行"""
        try:
            result = await asyncio.wait_for(
                asyncio.to_thread(agent_fn, state),
                timeout=self.config.agent_timeout_seconds
            )
            return {"success": True, "result": result, "agent": agent_name}
        except asyncio.TimeoutError:
            self.error_log.append({
                "agent": agent_name,
                "error": "timeout",
                "message": f"超过 {self.config.agent_timeout_seconds}s 限制"
            })
            return {"success": False, "error": "timeout", "agent": agent_name}
        except Exception as e:
            self.error_log.append({
                "agent": agent_name,
                "error": str(e)
            })
            return {"success": False, "error": str(e), "agent": agent_name}
    
    async def execute_with_retry(self, agent_fn, state: dict, 
                                  agent_name: str) -> dict:
        """带重试的 Agent 执行"""
        last_error = None
        for attempt in range(self.config.retry_count + 1):
            result = await self.execute_with_timeout(agent_fn, state, agent_name)
            if result["success"]:
                result["retry_attempt"] = attempt
                return result
            last_error = result
            if attempt < self.config.retry_count:
                await asyncio.sleep(self.config.retry_delay_seconds)
        last_error["retry_exhausted"] = True
        return last_error
```

### 5.8 并行执行组与人工中断

#### 5.8.1 parallel_groups 设计

`parallel_groups` 字段支持在工作流中并行执行多个 Agent，适用于互不依赖的子任务：

**配置示例**：
```json
{
  "parallel_groups": [
    {
      "group_id": "pg1",
      "agents": [1, 2],
      "merge_strategy": "concat",
      "timeout_seconds": 120
    }
  ]
}
```

| 字段 | 说明 |
|------|------|
| group_id | 并行组唯一标识 |
| agents | 组内 Agent ID 列表 |
| merge_strategy | 结果合并策略：`concat`（拼接）/ `first`（取首个）/ `merge_by_key`（按键合并） |
| timeout_seconds | 并行组总超时（取组内最慢 Agent 完成时间） |

**执行流程**：
```
用户输入 → [Agent A, Agent B] 并行执行
              ├─ Agent A ──┐
              └─ Agent B ──┼──▶ 结果合并 → 下一节点
                           (两者都完成或超时)
```

**实现代码**：
```python
import asyncio
from typing import List, Dict, Any

async def execute_parallel_group(
    agents: List[tuple],  # [(name, node_fn), ...]
    state: Dict[str, Any],
    timeout: int = 120
) -> Dict[str, Any]:
    """并行执行一组 Agent"""
    tasks = []
    for agent_name, node_fn in agents:
        task = asyncio.create_task(
            asyncio.to_thread(node_fn, state),
            name=agent_name
        )
        tasks.append(task)
    
    # 等待所有完成或超时
    done, pending = await asyncio.wait(
        tasks, timeout=timeout
    )
    
    # 收集结果
    results = {}
    for task in done:
        agent_name = task.get_name()
        if task.exception():
            results[agent_name] = {"error": str(task.exception())}
        else:
            results[agent_name] = task.result()
    
    # 未完成的标记为超时
    for task in pending:
        task.cancel()
        results[task.get_name()] = {"error": "timeout"}
    
    return {"parallel_results": results}
```

#### 5.8.2 human_interrupts 设计

`human_interrupts` 定义工作流中需要人工确认的中断点，适用于敏感操作（数据库写入、部署、配置变更等）。

**配置示例**：
```json
{
  "human_interrupts": [
    {
      "interrupt_id": "confirm_db_write",
      "agent_id": 3,
      "trigger_on": "before_execute",
      "message_template": "Agent {{agent_name}} 即将执行数据库写入操作，参数: {{params}}，是否继续？",
      "timeout_minutes": 30,
      "default_action": "reject"
    }
  ]
}
```

| 字段 | 说明 |
|------|------|
| interrupt_id | 中断点唯一标识 |
| agent_id | 触发中断的 Agent |
| trigger_on | 触发时机：`before_execute` / `on_tool_call` / `on_condition` |
| message_template | 提示消息模板（支持 `{{变量}}` 替换） |
| timeout_minutes | 等待人工决策超时时间 |
| default_action | 超时后的默认动作：`approve` / `reject` |

**执行流程**：
```
Agent 准备执行 → 命中 interrupt 规则
                  │
                  ▼
         ┌────────────────────┐
         │ 工作流暂停          │
         │ 通知用户确认        │
         │ 等待人工决策...     │
         └────────┬───────────┘
                  │
          ┌───────┴───────┐
          ▼               ▼
      [approve]       [reject]
    继续执行          跳过该步骤
```

**API 支持**：
```json
// POST /workflows/{id}/interrupts/{interrupt_id}/resolve
{
  "action": "approve",     // approve / reject
  "comment": "已确认操作安全"
}
```

### 5.10 工作流执行工作空间

#### 5.10.1 设计原则

每个工作流执行实例分配一个独立工作空间，所有参与 Agent 共享——不是 SDLC 专属能力，而是所有应用类型的通用基础设施。

- **通用性**：无论是研发 pipeline、线上问题排查还是代码审查，都使用同一套 workspace 机制
- **接力模式**：上游 Agent 产出 → 写入 workspace → 下游 Agent 读取，持续接力
- **两层上下文**：Prompt 传递摘要/决策 → 文件系统传递大体积产物（代码、报告、日志）

```
                ┌─────────────────────────────────────┐
                │         Workflow Workspace          │
                │  /data/workflow_workspaces/{exec}/  │
                │                                     │
                │  repos/          ← 共享代码仓库     │
                │  artifacts/      ← 各阶段产物目录    │
                │  context.json    ← 当前进度+摘要     │
                │  execution.log   ← 全流程日志        │
                └─────────────────────────────────────┘
                          ▲        │        ▲
                  ┌───────┘        │        └───────┐
                  ▼                ▼                ▼
             Agent A           Agent B           Agent C
          (读代码+写设计)  (读设计+改代码)   (读代码+跑测试)
```

#### 5.10.2 目录结构

```
/data/workflow_workspaces/{execution_id}/
│
├── repos/                           # 可选：共享代码仓库
│   └── {repo_name}/                 # Engine 负责 clone
│
├── artifacts/                       # Agent 产出物（按阶段隔离）
│   ├── 1_{first_agent}/
│   ├── 2_{second_agent}/
│   └── ...
│
├── context.json                     # 执行状态 + 上游摘要
└── execution.log                    # 结构化日志
```

**是否 clone 代码仓库取决于应用类型**：

| 应用类型 | repos/ | Agent 做什么 |
|---------|--------|-------------|
| SDLC 研发流程 | clone | designer 读代码做设计，developer 改代码，tester 跑测试 |
| 线上问题排查 | 不 clone | 分析 agent 读知识库，运维 agent 调 MCP 查 K8s/DB |
| 代码审查 | clone | reviewer agent 读代码，检查规范、安全、性能 |
| CI/CD 部署 | clone | builder agent 编译，deployer agent 发布 |

#### 5.10.3 context.json 协议

工作流执行期间唯一的状态文件，Agent 读取它了解上游发生了什么，Engine 通过它调度下一阶段。

```json
{
  "execution_id": "exec_abc123",
  "workflow_name": "线上问题排查-告警响应",
  "current_stage": 2,
  "status": "running",
  "upstream_summary": [
    {
      "stage": 1,
      "agent": "problem_analyzer",
      "status": "success",
      "summary": "告警来自 db-prod-03 实例连接数飙升，初步判断为慢查询堆积导致",
      "artifacts": ["analysis_report.md"]
    }
  ]
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| execution_id | string | 唯一执行 ID |
| workflow_name | string | 工作流名称 |
| current_stage | int | 当前执行到第几步（0-based） |
| status | string | running / completed / failed / paused |
| upstream_summary | array | 已完成阶段的摘要列表，供下游 Agent 快速了解上下文 |
| upstream_summary[N].agent | string | Agent 名称 |
| upstream_summary[N].summary | string | 1-2 句话说明该阶段做了什么和结论 |
| upstream_summary[N].artifacts | array | 产出物文件名列表（相对 artifacts/N_agent/） |

#### 5.10.4 Engine 职责

```python
class WorkflowWorkspaceManager:
    """工作流工作空间管理器 — 所有应用类型通用"""

    BASE_PATH = "/data/workflow_workspaces"

    async def setup(self, execution_id: str, repo_url: str = None) -> str:
        """
        Engine 启动工作流时调用，准备空间。
        是否 clone 仓库取决于应用类型，Engine 从 app 配置中获知。
        """
        ws = f"{self.BASE_PATH}/{execution_id}"
        os.makedirs(f"{ws}/artifacts", exist_ok=True)

        if repo_url:
            repo_dir = f"{ws}/repos/{self._repo_name(repo_url)}"
            await self._clone(repo_url, repo_dir)

        self._write_context(ws, {
            "execution_id": execution_id,
            "current_stage": 0,
            "status": "running",
            "upstream_summary": []
        })
        return ws

    async def stage_transition(self, ws: str, agent_name: str,
                                stage_num: int, result: dict):
        """一个阶段完成，更新 context.json，Engine 决定下一阶段"""
        context = self._read_context(ws)
        context["current_stage"] = stage_num + 1
        context["upstream_summary"].append({
            "stage": stage_num,
            "agent": agent_name,
            "status": result.get("status", "success"),
            "summary": result.get("summary", ""),
            "artifacts": result.get("artifacts", [])
        })
        self._write_context(ws, context)

    def build_agent_prompt(self, ws: str, agent_config: dict,
                           task_description: str) -> str:
        """为 Agent 构建包含 workspace 上下文的 System Prompt"""
        context = self._read_context(ws)

        summary_text = ""
        for s in context.get("upstream_summary", []):
            summary_text += f"- [{s['agent']}] {s['summary']}\n"
            if s.get("artifacts"):
                summary_text += f"  产出物: {', '.join(s['artifacts'])}\n"

        stage_dir = f"{context['current_stage'] + 1}_{agent_config['name']}"

        return f"""## 你的角色
{agent_config.get('system_prompt', '')}

## 当前任务
{task_description}

## 工作空间路径
{ws}

## 上游阶段摘要
{summary_text or '(无，这是第一个阶段)'}

## 工作空间说明
- 上游产物在 {ws}/artifacts/ 目录下，请先阅读了解上下文
- 如有代码仓库，在 {ws}/repos/ 目录下
- 你的产出物请写入 {ws}/artifacts/{stage_dir}/
- 完成后生成一个 summary（1-2句话），供下游 Agent 了解你这步做了什么
"""
```

#### 5.10.6 与 Agent 个人工作目录的关系

| | 工作流 workspace | Agent 个人 workspace |
|---|---|---|
| 路径 | `/data/workflow_workspaces/{exec}/` | `/data/agent_workspaces/agent_{id}/` |
| 生命周期 | 一次执行（执行结束可归档或清理） | 持久存在（Agent 被删除时清理） |
| 用途 | 流程中共享代码和产物 | Agent 专属配置（CLAUDE.md、Skills） |
| 谁创建 | WorkflowEngine | AgentWorkspaceManager |
| 内容 | repos/ + artifacts/ + context.json | CLAUDE.md + 个人配置 |

Agent 执行时，工作目录指向个人 workspace，同时被注入工作流 workspace 路径作为共享上下文。

#### 5.10.7 清理策略

| 时机 | 动作 |
|------|------|
| 工作流成功完成 | 归档保留 7 天，随后清理 |
| 工作流失败 | 保留 workspace 供调试，30 天后清理 |
| 手动触发 | 管理员可通过 API 清理指定执行的工作空间 |
| 磁盘告警 | 后台任务监控磁盘使用率，超过 80% 时优先清理已完成的旧 workspace |
