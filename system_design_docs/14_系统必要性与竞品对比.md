# 系统必要性与竞品对比

## 一、系统简介

**研发团队多Agent协同平台 (hh-agent-hub)** 是一个面向研发团队的内部多 Agent 协同平台，核心理念是：

> 将研发工具链（MCP Server）、团队知识（知识库）、标准化流程（Skill）统一注册为**平台级资源**，研发人员通过**配置 Agent + 编排工作流**快速搭建智能助手应用，覆盖从问题排查到需求交付的全场景。

一句话总结：**用 Agent 编排替代人工在多系统之间切换，让研发流程可自动化、可追踪、可复用。**


## 二、要解决的真实问题

研发团队的日常不是"缺工具"，而是**工具太多、切换太频繁、流程不可见**：

| 场景 | 现状 | 本系统方案 |
|------|------|-----------|
| 线上告警排查 | 1人看监控 → 切到数据库查慢查询 → 切到 K8s 看 Pod → 切到 Wiki 查 SOP → 写总结 | 配置"告警排查"应用 → 问题分析 Agent 读知识库 → 运维 Agent 调 MCP 查 DB/K8s → 自动汇总报告 |
| 需求开发上线 | 需求文档 → 开会 → 开发 → 找人 Review → 测试 → 找运维部署 → 每一步靠人推进 | 需求分析 Agent → 方案设计 Agent → 开发 Agent → 审查 Agent → 测试 Agent → CI/CD Agent → 全流程可追踪 |
| 知识沉淀 | 老员工脑子里的排查经验 → 离职后丢失 | 排查过程沉淀为 Skill 模板 → 新人也能复用 |
| 权限安全 | DBA 怕 Agent 执行 DELETE → 不敢给工具权限 | 工具级授权 + 人工中断确认 + 完整审计日志 |

## 四、为什么不自研而直接用 Dify/Coze？

Dify 和 Coze 是优秀的通用 AI 应用平台，但面向的是"搭建一个 ChatBot"场景。研发团队的场景完全不同：

1. **不是对话应用，是流程应用**：Dify 的"工作流"是在一个对话内按步骤执行，不适合"需求→设计→开发→测试→部署"这种跨角色、跨工具、耗时数十分钟的流程编排

2. **研发工具体系是 MCP，不是 API**：研发团队已有的数据库、K8s、监控系统通过 MCP 协议暴露，Dify 不支持 MCP Server 的资源目录管理和工具发现

3. **工具授权需要细粒度控制**：DBA 允许 Agent 查表但不能删表，运维允许查 Pod 状态但不能重启——这种工具级授权在通用平台上是缺失的

4. **需要 Claude Code 深度集成**：代码开发类 Agent 需要文件读写、命令执行、Git 操作能力，这是 Claude Code 的强项，Dify/Coze 的本地 Agent 做不到

5. **数据不出内网**：研发代码、数据库 schema、告警日志都不能离开公司网络，必须自托管


## 五、系统必要性的四个"如果"

| 如果没有本系统 | 后果 |
|------------|------|
| 工具切换 | 查一个问题，需要手动登录 3-4 个系统，复制粘贴数据，耗时 15-30 分钟 |
| 知识流失 | 老员工的排查经验随离职带走的比例 ≈ 100% |
| 流程推动 | 需求到上线的每一步靠人在群里催，没有自动化接力 |
| Agent 碎片化 | 每个团队各自搞一套 Agent，工具和 Skill 无法复用 |

## 六、一句话总结

> 研发团队不缺少工具，缺少的是把工具、知识、流程串联成可复用的自动化应用的平台。hh-agent-hub 就是做这件事的。
