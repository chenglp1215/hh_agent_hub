# 系统测试方案

## 一、测试策略概述

### 1.1 测试阶段

```
阶段 0: 单元测试      → 每个 Agent/MCP/Workflow 模块独立验证
阶段 1: 单 Agent 测试  → 单独验证每种类型 Agent 能否正确执行任务
阶段 2: 工作流测试     → 验证多 Agent 协作编排的正确性
阶段 3: 应用集成测试   → 模拟真实场景端到端验证
阶段 4: 性能与压力测试  → 验证并发能力和稳定性
```

### 1.2 测试环境

| 环境 | 用途 | 数据库 |
|------|------|--------|
| dev | 开发期间自测 | SQLite 本地文件 |
| test | 自动化测试 CI | SQLite 内存模式 |
| staging | 预发布验证 | SQLite + WAL 模式 |
| prod | 生产 | SQLite → PostgreSQL |

### 1.3 Mock 策略

| 组件 | Mock 方式 | 说明 |
|------|---------|------|
| LLM 响应 | `unittest.mock` | 避免调用真实 API，确保测试快速稳定 |
| MCP Server | `MockMCPServer` | 自建的 JSON-RPC 模拟服务器 |
| 知识库检索 | 注入固定检索结果 | 不依赖实际文件系统 |
| Claude Code CLI | Mock 子进程输出 | 不依赖实际 Claude Code 环境 |

## 四、阶段 2：工作流测试

### 4.1 监督者模式

| 用例编号 | 场景 | 前置条件 | 预期 |
|---------|------|---------|------|
| WF-001 | 主管路由到 Worker A | 主管 Agent 返回 next_agent="agent_a" | agent_a 被执行 |
| WF-002 | 主管判定结束 | 主管返回 next_agent="end" | 工作流正常结束 |
| WF-003 | Worker 返回主管 | agent_a 执行完毕 | 回到 supervisor 等待下一步路由 |

### 4.2 顺序模式

| 用例编号 | 场景 | 前置条件 | 预期 |
|---------|------|---------|------|
| WF-101 | 3 个 Agent 顺序执行 | Agent 1→2→3 | 按 1→2→3 顺序执行 |
| WF-102 | 中间 Agent 失败 + fail_fast | Agent 2 抛异常 | Agent 3 不执行，返回 Agent 1 的结果 |
| WF-103 | 中间 Agent 失败 + skip_continue | Agent 2 抛异常 | Agent 3 仍然执行 |

### 4.3 重试策略

| 用例编号 | 场景 | 前置条件 | 预期 |
|---------|------|---------|------|
| WF-201 | 第 2 次重试成功 | Agent 前 1 次失败 | retry_attempt=1, success=True |
| WF-202 | 全部重试失败 | Agent 连续 3 次失败 | retry_exhausted=True, success=False |

### 4.4 并行组

| 用例编号 | 场景 | 前置条件 | 预期 |
|---------|------|---------|------|
| WF-301 | 两个 Agent 并行完成 | Agent A 3s, Agent B 1s | 总耗时 ≈ 3s (取最长) |
| WF-302 | 一个超时 | Agent A 正常, Agent B 超时 | Agent A 结果正常，Agent B 标记 timeout |

### 4.5 人工中断

| 用例编号 | 场景 | 前置条件 | 预期 |
|---------|------|---------|------|
| WF-401 | 触发中断并暂停 | before_execute 规则命中 | 工作流状态变为 paused |
| WF-402 | 人工 Approve | POST /resolve {action:"approve"} | 继续执行 |
| WF-403 | 人工 Reject | POST /resolve {action:"reject"} | 跳过该步骤 |
| WF-404 | 超时默认动作 | 超时 30 分钟 | 按 default_action 处理 |

### 4.6 工作空间接力

| 用例编号 | 场景 | 预期 |
|---------|------|------|
| WF-501 | 上游产物被下游读取 | Agent 2 能读到 artifacts/1_xxx/ 的内容 |
| WF-502 | context.json 正确传递 | Agent 3 的 Prompt 包含 Agent 1 和 Agent 2 的 summary |
| WF-503 | repo 共享 | Agent 2 的修改对 Agent 3 可见 |


## 五、阶段 3：场景集成测试

### 5.1 场景 A：线上问题排查

```
配置:
  App: "告警响应助手"
  Workflow: sequential, 2 个 Agent
    - problem_analyzer (local, KB: 运维手册)
    - ops_investigator (local, MCP: mysql-prod, k8s-cluster)

测试用例:
```

| 编号 | 输入 | 预期流程 | 验证点 |
|------|------|---------|--------|
| SCN-A01 | "db-prod-03 CPU 90% 告警" | analyzer 读知识库分析 → investigator 调 MCP 查库查 Pod → 汇总报告 | 1) 知识库检索命中 2) MCP 工具被调用 3) 最终报告含具体原因 |
| SCN-A02 | "服务 502 错误" | analyzer 从知识库匹配"网关超时"SOP → investigator 查 Nginx 日志 + Pod 状态 | trace.json 包含完整调用链 |
| SCN-A03 | 知识库无匹配 | analyzer 无法从 KB 匹配到相关 SOP | "无法确定原因，建议人工介入" |

### 5.2 场景 B：SDLC 需求到交付

```
配置:
  App: "需求交付助手"
  Workflow: sequential + parallel_group, 4 个 Agent
    - designer (claudecode, repo: team/myapp)
    - developer (claudecode)
    - tester (claudecode)
    - cicd (local, MCP: k8s-cluster)
  中断点: cicd 执行前需人工确认
```

| 编号 | 输入 | 预期流程 | 验证点 |
|------|------|---------|--------|
| SCN-B01 | "添加用户登录日志功能" | designer 输出方案 → developer 改代码 → tester 跑测试 → cicd 等待确认 | 1) designer 产物可读 2) developer 修改了 repos/ 下的文件 3) tester 能在修改后的代码上运行 4) 人工中断生效 |
| SCN-B02 | tester 发现测试失败 | designer → developer → tester 返回 2 个失败 | workflow 状态=partial，trace 记录失败详情 |
| SCN-B03 | 人工拒绝部署 | cicd 中断 → 用户 reject | 部署被跳过，工作流标记完成 |

### 5.3 场景 C：代码审查

```
配置:
  App: "代码审查助手"
  Workflow: supervisor, 2 个 Worker
    - code_reviewer (claudecode)
    - security_checker (claudecode)
  Supervisor 根据 PR 类型决定调用哪个 Worker
```

| 编号 | 输入 | 预期流程 | 验证点 |
|------|------|---------|--------|
| SCN-C01 | "审查 PR #123" | supervisor → reviewer 检查代码规范 + security 检查安全 | 两个 Agent 的结果都汇总 |
| SCN-C02 | "安全审查 PR #124" | supervisor 只路由到 security_checker | reviewer 不被调用 |

## 八、CI/CD 集成

```yaml
# .github/workflows/test.yml (示例)
name: System Test

on: [push, pull_request]

jobs:
  unit-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - run: pytest tests/unit/ -v --cov=backend --cov-report=xml

  integration-test:
    needs: unit-test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest tests/integration/ -v --run-mcp-mock

  scenario-test:
    needs: integration-test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest tests/scenarios/ -v --run-slow
```

## 九、测试覆盖率目标

| 层级 | 覆盖率目标 | 说明 |
|------|----------|------|
| 核心模块 (agent_factory, workflow_engine, mcp_client) | ≥ 85% | 关键路径必须充分覆盖 |
| API 端点 | ≥ 80% | 所有端点至少一个 happy path + 一个 error case |
| Agent 节点 | ≥ 75% | 每种 Agent 类型的核心逻辑 |
| 前端组件 | ≥ 60% | 关键交互（Agent 编辑、工作流画布） |
| 整体 | ≥ 70% | 项目整体覆盖率 |
