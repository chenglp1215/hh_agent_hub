## 修改概述
将系统设计文档中的数据库从 PostgreSQL 改为 SQLite，以实现轻量化部署，无需独立的数据库服务。

- 时间戳：`DEFAULT NOW()` → `DEFAULT CURRENT_TIMESTAMP`

#### 3.6 sessions 表
- 主键：保持 `VARCHAR(36) PRIMARY KEY`

- JSON字段：`JSONB` → `TEXT`
- 外键：`INT` → `INTEGER`
- 时间戳：`DEFAULT NOW()` → `DEFAULT CURRENT_TIMESTAMP`

## SQLite 优势

1. **零配置**: 无需安装和配置独立的数据库服务
2. **轻量级**: 单个文件存储，占用资源少
3. **易备份**: 直接复制文件即可备份
4. **易迁移**: 数据库文件可在不同环境间直接复制
5. **性能优秀**: 对于中小型应用，性能表现良好

## SQLite 注意事项

1. **并发写入**: SQLite 并发写入能力有限，适合读多写少场景
2. **数据库文件**: 需要确保数据目录权限正确
3. **数据类型**: SQLite 使用动态类型系统，字段类型约束较弱
4. **JSON处理**: 使用 TEXT 存储 JSON，应用层需自行处理序列化/反序列化
5. **布尔类型**: SQLite 使用 0/1 表示 false/true

## 兼容性处理

应用代码中需要注意：
```python
# JSON字段处理示例
import json

# 存储时序列化
agent.mcp_servers = json.dumps(mcp_server_list)

# 读取时反序列化
mcp_server_list = json.loads(agent.mcp_servers)

# 布尔值处理
app.enabled = 1  # True
app.enabled = 0  # False
```

## 后续建议

1. 对于高并发场景，可考虑使用 SQLite 的 WAL 模式：
   ```sql
   PRAGMA journal_mode=WAL;
   PRAGMA synchronous=NORMAL;
   ```

2. 定期执行 VACUUM 命令优化数据库：
   ```sql
   VACUUM;
   ```

3. 如果未来需要升级到 PostgreSQL，迁移脚本简单：
   - 使用 `pgloader` 工具
   - 或编写 Python 脚本读取 SQLite 数据插入 PostgreSQL


## v1 开发策略（2026-06 确定）

### SQLite 定位

| 阶段 | 数据库 | 目标并发 | 说明 |
|------|--------|---------|------|
| **v1 验证期** | SQLite + WAL 模式 | 20-30 会话 | 功能跑通验证，小团队内部使用 |
| **v2 扩展期** | PostgreSQL | 100+ | 多团队使用，高并发场景 |

### v1 代码层准备

使用 SQLAlchemy 作为 ORM 层，确保切换数据库时只需修改连接字符串：

```python
# v1: SQLite
DATABASE_URL = "sqlite+aiosqlite:////data/agent_platform.db"

# v2: PostgreSQL（仅改环境变量）
DATABASE_URL = "postgresql+asyncpg://user:pass@host:5432/agent_platform"
```

### v1 WAL 模式优化

```sql
-- 启动时执行
PRAGMA journal_mode=WAL;       -- 提升并发读取
PRAGMA synchronous=NORMAL;     -- 平衡性能与安全
PRAGMA cache_size=-64000;      -- 64MB 缓存
PRAGMA busy_timeout=5000;      -- 5秒忙等
PRAGMA foreign_keys=ON;        -- 启用外键约束
```
